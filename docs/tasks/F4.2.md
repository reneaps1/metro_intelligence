## 🎯 Objetivo

Implementar autenticación (argon2, JWT access+refresh) y autorización RBAC deny-by-default como dependency de FastAPI, aplicando la matriz de `docs/security/rbac.md`, con rate limiting y lockout en `/auth`.

## 📋 Contexto

**Esta es la tarea de seguridad más crítica del MVP.** Ningún endpoint posterior se mergea sin usar la dependency `require_permission(...)` que esta tarea crea. El demo usa usuarios locales; una fase futura (F11.1, fuera de alcance del demo) añadirá SSO sin cambiar el contrato.

## ✅ Alcance

- `POST /auth/login`, `/auth/refresh`, `/auth/logout`; hashing argon2id; JWT HS256 con `exp`/`iat`/`jti`.
- Dependency `require_permission(resource, action)` que lee roles→permisos de la BD (ya sembrados por la migración `0001_org_security` — ver `ROLE_PERMISSIONS` en `backend/alembic/versions/0001_org_security.py` para la lista completa de tokens de permiso); **deny by default**.
- Lockout tras N intentos fallidos + rate limiting (slowapi u otra lib) en `/auth/*`.
- Auditoría de login exitoso/fallido usando `security_audit_log` (append-only, ya existe — ver el trigger en `0001_org_security.py`).
- `GET /me`.

## 🚫 Fuera de alcance

SSO/LDAP/MFA (fase futura); gestión de usuarios admin (eso es parte de F4.4).

## Referencias obligatorias

## ✔️ Criterios de aceptación

- [ ] Un test parametrizado rol×endpoint×acción cubre **toda** la matriz de `rbac.md` (200/403 para cada combinación).
- [ ] JWT forjado o expirado → 401 (tests explícitos).
- [ ] El 6º intento fallido de login → lockout + entrada de auditoría.
- [ ] Cero secretos hardcodeados; `JWT_SECRET` viene de env (documentado en `.env.example`).
