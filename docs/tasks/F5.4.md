## Estado de dependencias

- **F4.2 (JWT auth backend):** ❌ Bloqueador directo — necesitas un backend real de auth para conectar esto.

## ⚠️ Importante: reemplaza, no construyas desde cero

`frontend/src/lib/auth/AuthProvider.tsx` y `frontend/src/features/auth/LoginPage.tsx` (F5.M) ya existen, pero son **completamente mockeados**: el login es un selector de rol sin credenciales reales, sin JWT, sin llamadas de red. Esta tarea reemplaza esa implementación por una real, manteniendo la misma interfaz pública (`useAuth()` con `user`, `login()`, `logout()`) para que el resto del frontend (que ya consume ese hook en `Topbar.tsx`, `RiskPage.tsx`, `ImportPage.tsx`, `App.tsx`) no tenga que cambiar.

## 🎯 Objetivo

Página de login real (validación on-blur, errores claros, estados de carga), API client con manejo de access/refresh tokens y reintento transparente, y route guards que redirigen según sesión/rol.

## 📋 Contexto

Contrato de F4.2. **Access token en memoria, nunca en localStorage** — el mock actual sí guarda el usuario en `localStorage` (aceptable para un mock sin credenciales reales, pero esto NO debe sobrevivir a esta tarea para el token real). Refresh vía cookie httpOnly si el backend lo soporta, o rotación en memoria — decisión a coordinar con quien hizo F4.2.

## ✅ Alcance

- `/login` con el branding del manual, ambos temas (el mock ya tiene la UI visual — reutilízala, cambia la lógica de submit).
- `frontend/src/lib/api.ts` (nuevo): interceptores, refresh automático, logout en refresh fallido, mapeo de errores a mensajes de usuario (sin exponer internals del backend).
- Guards: ruta protegida sin sesión → redirect a login (el mock ya tiene un `RequireAuth` básico en `App.tsx` — revísalo, probablemente sirve de base); ruta sin permiso → 403/404 amigable.
- Contexto de sesión poblado desde `GET /me` (F4.2).

## 🚫 Fuera de alcance

SSO (fuera del demo); registro de usuarios (eso lo hace un admin vía F4.4).

## ✔️ Criterios de aceptación

- [ ] Login/logout/expiración manejados sin estados "zombie" (sesión que parece activa pero no lo está).
- [ ] El token de acceso **nunca** toca `localStorage` (revisión de código + test).
- [ ] Errores 401/403/500 muestran mensajes limpios al usuario, sin jerga técnica ni stack traces.
- [ ] El guard por rol está verificado (ej. `viewer` intentando acceder a una ruta de admin → 403).
