## 🎯 Objetivo

Exponer el catálogo maestro vía REST: CRUD de familias, partes, características, especificaciones (con **versionado explícito**: crear nueva versión, nunca sobrescribir), programas de medición, planes de inspección; más administración básica de usuarios (solo admin).

## 📋 Contexto

Este es el contrato que consumirá el frontend de catálogo (F5.5 — que ya existe como mock en `frontend/src/features/catalog/`, ver abajo). Cambiar una tolerancia = crear una nueva versión de spec con `valid_from`; la anterior se cierra con `valid_to`. Todo cambio auditado vía F4.3.

## ✅ Alcance

- Routers con schemas Pydantic tipados request/response, paginación, filtros.
- `require_permission` (de F4.2) en cada endpoint según `docs/security/rbac.md`.
- Endpoint específico `POST /characteristics/{id}/specifications` que crea una nueva versión (nunca un UPDATE de la spec activa — la migración `0002_catalog.py` tiene un índice único parcial `valid_to IS NULL` por característica que lo hace cumplir a nivel de DB, tu API solo necesita respetarlo).
- Auditoría vía F4.3 en toda escritura.
- OpenAPI etiquetada y coherente — es el contrato real para el frontend.

## 🚫 Fuera de alcance

Importación de catálogo desde archivos (post-demo); la UI en sí (F5.5).
