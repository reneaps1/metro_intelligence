## 🎯 Objetivo

Suite que valida el seed (constraints, sanidad estadística, lint de confidencialidad) + archivos CSV/Excel de muestra que se importarán **en vivo** durante el demo.

## 📋 Contexto

CLAUDE.md §11/§20: ningún dato debe parecerse a un identificador real de cliente. Este lint es el gate de CI y **release blocker** — es, literalmente, el control de confidencialidad del proyecto entero.

## ✅ Alcance

- `tests/seed_validation/`: constraints (balloon únicos, specs consistentes, etc. — mayormente ya los validan los tests de migración, pero aquí se corre contra el dataset generado completo), rangos estadísticos por escenario (reutiliza el enfoque de `seed/tests/test_measurement_series_generator.py`), y **lint de confidencialidad**: rechaza patrones de números de parte reales de OEMs conocidos, nombres de plantas reales, emails que no sean `.local`.
- `seed/sample_files/`: 2-3 CSV y 1 Excel con mediciones **posteriores** al histórico de 90 días (para que al importarlas en vivo durante el demo se vea: import → evaluación → alerta, en tiempo real, no datos ya vistos).
- Lint integrado a CI (si `.github/workflows/ci.yml` ya existe cuando hagas esto; si no, dócumentalo para cuando F1.2 lo cree).

## 🚫 Fuera de alcance

El pipeline de importación en sí (eso es F4.5) — aquí solo generas los archivos de muestra, no los parseas contra un backend real.
